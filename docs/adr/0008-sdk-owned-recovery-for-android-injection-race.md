# SDK-owned recovery for the Android release-build injection race

Expo DOM components depend on globals (`$$EXPO_DOM_HOST_OS`, `$$EXPO_INITIAL_PROPS`) that native injects via `injectedJavaScriptBeforeContentLoaded`. On Android, react-native-webview delivers that script through `onPageStarted` + `evaluateJavascript`, which is not guaranteed to run before page scripts when the DOM bundle loads from `file:///android_asset` (expo/expo#41798). Release builds load fast enough to lose this race; dev builds never lose it (Metro HTTP latency delays page scripts) and iOS injects synchronously via `WKUserScript`. A lost race has two permanent consequences: expo's `dom-entry` throws before mounting the React root, and expo's `marshal` locks `IS_DOM = false` at module scope, so its bridge (`emit`, prop updates, native action proxies, the `matchContents` body observer) stays dead even after the injection lands hundreds of milliseconds later.

Patching expo in the consuming app fixes the symptom but not the product: every consumer of this SDK would have to carry the same patch in their own Expo install. Because DOM component web bundles are compiled from SDK-shipped source inside the consumer's app build, a fix that lives in our source lands in every consumer's WebView automatically. The SDK therefore owns the recovery.

[`packages/ui/src/lib/dom-bridge-recovery.tsx`](../../packages/ui/src/lib/dom-bridge-recovery.tsx) evaluates before expo's generated DOM entry calls `registerDOMComponent` (the entry requires the component module as the call's argument). When it detects a lost race — `ReactNativeWebView` present but `$$EXPO_DOM_HOST_OS` missing — it:

1. defines placeholder globals so `dom-entry` mounts instead of throwing,
2. traps the late injection with a `$$EXPO_INITIAL_PROPS` property setter to capture marshalled props, and
3. rebuilds the bridge on the raw transport primitives that are **not** gated by `IS_DOM`: `window.ReactNativeWebView.postMessage` (web → native) and the `$$dom_event` CustomEvents native dispatches through `injectJavaScript` (native → web). This covers prop updates (native re-emits `$$props` on every native render), native action calls (`$$native_action` / `$$native_action_result`), and `matchContents` sizing (`$$match_contents_event`, which native ignores unless the `matchContents` dom prop is set).

Every DOM component exports through `withDomBridgeRecovery`. When the race is won (dev builds, iOS, web), the HOC returns the component unwrapped and the module is inert — zero runtime cost. The wire constants in this module mirror `expo/src/dom/injection.ts` and must match exactly; revisit them when upgrading Expo, and retire this module entirely if expo ships an injection mechanism that cannot lose the race.

Layer-1 tests live in [`packages/ui/src/lib/__tests__/dom-bridge-recovery.test.ts`](../../packages/ui/src/lib/__tests__/dom-bridge-recovery.test.ts). The recovery path itself was validated on-device (Samsung A15, Android 14, `expo run:android --variant release`): reader render, version/chapter picker commits, settings round trip, footnote sheet with `matchContents` sizing, VerseOfTheDay, and BibleCard all function after a lost race.
